**Phase 3: DO migration (isidora)**

Strategy: bridge-and-migrate using `docker network connect --alias`.

1. **Backup**: pg_dumpall (193MB), copied to local machine, database list + volume list snapshot
2. **Bridge**: Created `infra` network, connected old postgres/redis/traefik with aliases (`postgres`, `redis`)
3. **Migrated apps one at a time** (marie → vaultwarden → newsintel → company-intel → caitie):
   - Updated compose files: networks → `infra`, labels → `traefik.docker.network=infra`
   - Updated .env files: hostnames `caitie_postgres_1` → `postgres`, `caitie_redis_1` → `redis`
   - Company-Intel had hardcoded hostnames in compose env section — updated inline
   - Vaultwarden: moved ADMIN_TOKEN from hardcoded in compose to .env
   - Caitie: compose + env updated but app left down (already exited for 2+ days)
4. **Replaced infra containers**: stopped old infra, removed old networks, started new infra, restarted all apps
5. **Cleanup**: removed stale directories (betty, gvoice, realtalk, vagent, iconic-video-player, home), stale networks, monitoring stack (Grafana, Prometheus, Loki, Promtail)

**Issues encountered during migration**

- **YAML escaping via SSH heredoc**: backticks in Traefik labels got mangled. Fix: write compose files locally, `scp` them to server
- **Docker Compose network label mismatch**: `docker network create infra` creates a network without Compose labels. When Compose tried to adopt it, it failed with "incorrect label". Fix: disconnect all containers, remove network, let Compose recreate it
- **Old infra `docker compose down` didn't fully stop**: the old compose had env var warnings (missing POSTGRES_PASSWORD in that shell), containers kept running. Fix: explicit `docker stop` + `docker rm`
- **Port conflict**: old redis still holding 6379 when new redis tried to start. Root cause: same as above (old containers not stopped)
- **Apps disconnected after network recreation**: when the `infra` network was removed and recreated, running app containers lost their network attachment. Fix: full `docker compose down && up` cycle for each app

**README updated**

- Langfuse setup documentation (first-time + per-project)

### Decisions made

- **pgvector on DO**: switched from `postgres:16-alpine` to `pgvector/pgvector:pg16` even though no databases currently use vector. Future-proofed — backward compatible.
- **Langfuse on all environments**: local, DO, and AWS will all run Langfuse for LLM traceability
- **Monitoring stack removed from DO**: Grafana, Prometheus, Loki were not used and not present on local. Consistency across environments.
- **Caitie stays down**: all 6 containers had been exited for 2+ days. Compose + env updated for when it's needed.
- **Company-Intel keeps HTTP entrypoint**: uses `entrypoints=web` (not websecure). Left as-is to avoid breaking existing routing — fix separately.
- **scp over SSH heredoc for compose files**: backtick escaping in YAML via SSH is unreliable. Write locally, transfer via scp.
