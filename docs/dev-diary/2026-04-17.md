## Session 13 — aws01 security audit + hardening

### What we worked on

Started as a "can I see Langfuse in the browser?" question and kept peeling back layers: Langfuse was running on aws01 bound to `127.0.0.1:3030` with no Traefik route; `dev.collectiveagencies.ai` resolves to the box but only path-routes `/marie` on port 80; and nothing is wired on 443. DNS pointed at the box but no TLS/certs were ever issued — the 443 listener existed and did nothing. Pivoted into a full security audit of the repo + aws01 runtime, then applied eight hardening fixes.

### Quick win first

User wanted Langfuse UI access without DNS changes. Recommended an SSH local port forward (`ssh -L 3030:localhost:3030 aws01`) → browse `http://localhost:3030`. No config change, no new routes. Simpler than wiring a subdomain and fighting Next.js `basePath` (not env-tunable in the upstream langfuse image without a rebuild).

### Audit findings (abbreviated)

**Repo:**
- `.env.example` shipped weak "production" defaults (`postgres/postgres`, `local-langfuse-secret-dev-only`, `adminadmin`) that silently become real secrets if copied unedited.
- Default binds at `0.0.0.0` for postgres/redis/langfuse/umami/dashboard.
- Images pinned to major-only tags (`langfuse:2`, `redis:7-alpine`, `umami:postgresql-latest`) — supply-chain exposure.
- `init-project.sh` has `eval "$@"` and unquoted `${DB_NAME}` in `psql -c`; theoretical command-injection shape if inputs are ever malicious.

**aws01 runtime:**
- `/app/infra/.env` at mode `664` (world-readable on the box).
- `ubuntu` user has `NOPASSWD: ALL` sudo (standard EC2 Ubuntu AMI default — worth knowing).
- `PermitRootLogin without-password`; no UFW; no fail2ban.
- Traefik `--api.insecure=true` (loopback-only, so low blast radius, but footgun).
- ACME email `noop@localhost`, `acme.json` 0 bytes — no certs ever issued, yet 443 exposed to `0.0.0.0` doing nothing.
- Umami crashlooping because its postgres database was never created (`database "umami" does not exist`).
- **Syxsense Responder** (managed RMM, runs as root, listens on `:51342`). Paused to confirm with user before flagging as suspicious — confirmed IT-authorized.

### What got done

**Quick wins:**
- `chmod 600 /app/infra/.env` on aws01; same locally.
- `docker exec postgres psql -U postgres -c 'CREATE DATABASE umami;'` + `docker restart umami` → healthy, migrations applied cleanly.

**Repo (committed `ea5ed92`):**
- Pinned digests on `traefik:v3.6`, `pgvector/pgvector:pg16`, `redis:7-alpine`, `langfuse/langfuse:2`, `umami:postgresql-latest`. Resolved via `docker image inspect` on aws01 (images already pulled). Vaultwarden + Mailpit skipped — not pulled anywhere queryable; pin when next activated.
- Gated 80/443 bindings behind `HTTP_BIND` / `HTTPS_BIND` env (default `0.0.0.0`, preserves Hetzner behavior).
- `.env.example`: all BIND defaults → `127.0.0.1`. Loud prod-checklist header. Softened the `adminadmin` default and added "DELETE these in prod" next to the `LANGFUSE_INIT_*` block.

**aws01:**
- Added `HTTPS_BIND=127.0.0.1` to `/app/infra/.env` → public 443 is now loopback-only until TLS is wired.
- `git pull` + `docker compose up -d` → recreated all services with new digests. ~2s traefik restart; marie HTTP 200 before and after.
- `/etc/ssh/sshd_config.d/99-hardening.conf` with `PermitRootLogin no`, `X11Forwarding no`, `MaxAuthTries 3`, `LoginGraceTime 30`. Validated with `sshd -t` before reload.
- `ufw --force reset` → defaults (deny in / allow out) → `allow 22/80/51342` (SSH, HTTP, Syxsense) → `--force enable`. Verified session stayed up and `dev.collectiveagencies.ai/marie` still returns 200.

### Decisions

- **Gate HTTPS via env var, don't delete from compose.** Hetzner uses 443; aws01 doesn't yet. One parameterized compose works for both.
- **Pin digests, not version tags.** Explicit request; supply-chain value is worth the maintenance cost.
- **Keep local-dev `.env.example` defaults working.** Devs expect `cp .env.example .env && start` to just work. Loud header warning beats forcing everyone to edit before first run.
- **Skipped fail2ban.** Password auth is off and `MaxAuthTries 3` covers what f2b would catch. Not worth the moving part.
- **Skipped `ufw-docker` integration.** Aware Docker's iptables rules bypass UFW for published ports. UFW still controls host-native listeners (22, 51342, anything future). Full integration is its own project.

### Issues encountered

- **Domain typo chase.** User reported "someone wired `dev.collectiveagencies.dev/langfuse` to aws01"; `.dev` wasn't registered at all. Spent a round on `whois`/`dig` before getting the correction to `.ai`. Real domain resolves to `100.64.0.3` from laptop (Tailscale CGNAT) and `10.251.8.172` from within VPC — something between the tailnet and VPC bridges the traffic. Didn't trace the exact path; not relevant to this session's work.
- **EC2 IMDS `public-ipv4` returned HTML 404** (not the standard EC2 404 body). Consistent with the box being VPC-internal (no public IP), but the response looks like a web-server 404. Worth asking the network owner about at some point; not blocking.
- **Traefik 443 was pure attack surface** — exposed on `0.0.0.0`, zero routers on `websecure`, `acme.json` empty. Fixed by binding loopback-only until TLS is actually wired.
- **Next.js `basePath` is build-time only.** Blocked the obvious "route `/langfuse` like `/marie`" path; the langfuse image doesn't expose the env var to avoid rebuilding. Noted for later if we ever need sub-path hosting.
