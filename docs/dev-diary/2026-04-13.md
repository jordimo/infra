## Session 1 — THECOLLECTIVE_AWS01 infrastructure from scratch

### What we worked on

Set up the full deployment infrastructure for THECOLLECTIVE_AWS01 (EC2 at 52.72.211.242) — from an empty VM with nothing installed to running Traefik, Postgres, and Redis. Also configured multi-account GitHub SSH on both dev and corporate laptops.

### What got done

**Infrastructure repo (`jordimo/THECOLLECTIVE_AWS_INFRA`, public)**

- Created `traefik/docker-compose.yml` — Traefik v3.2, pgvector/pg16, Redis 7 Alpine on a shared `traefik-public` Docker network. Postgres/Redis are internal only (no ports exposed to host). Traefik on port 80 + dashboard on 8080. Commit `ea9fd80`.
- Created `traefik/dynamic/middlewares.yml` — placeholder for shared Traefik middlewares.
- Created `traefik/.env.example` — template with `POSTGRES_PASSWORD`.
- Created `project.sh` — registers projects with Traefik using path-based routing (`/project-name` → web, `/project-name/api` → api). Generates dynamic Traefik config files with `stripPrefix` middleware per project. Commands: `add`, `rm`, `ls`.
- Created `setup.sh` — one-time VM setup: checks Docker, copies `.env.example`, creates `traefik-public` network.
- Created `start.sh` / `stop.sh` — start/stop infrastructure via `docker compose up -d` / `down`.
- Created `deploy.sh` — run from corporate laptop over SSH. Supports `./deploy.sh` (infra only), `./deploy.sh <project>` (specific project), `./deploy.sh --all`.
- Created `bootstrap.sh` — for a completely blank Ubuntu EC2: installs Docker CE + Compose plugin + Git, clones repo to `/app/Deployer`, runs setup. Commit `f06688c`.

**Key design decisions vs local-infra**

| Aspect | Local (`local-infra`) | VM (`THECOLLECTIVE_AWS_INFRA`) |
|---|---|---|
| Routing | Host-based (`joann.local`) | Path-based (`/joann`) |
| TLS | mkcert certs, HTTPS | HTTP only (ports 80/443 pending IT) |
| Network | `local-dev` | `traefik-public` |
| DNS | `/etc/hosts` entries | Public IP only |
| Exposed ports | Postgres 5432, Redis 6379 on host | Internal only |
| Middleware | `redirect-to-https` | `stripPrefix` per project |

**VM bootstrapping**

- Repo was initially private. Changed to public so the VM can `curl` the bootstrap script without needing GitHub credentials.
- Bootstrap command from corporate laptop: `ssh ubuntu@52.72.211.242 'curl -fsSL https://raw.githubusercontent.com/jordimo/THECOLLECTIVE_AWS_INFRA/main/bootstrap.sh | bash'`
- Hit Docker socket permission error after bootstrap — fixed by logging out/in to pick up `docker` group membership.
- Hit `traefik-public` network label mismatch — `setup.sh` created the network manually before Compose ran. Fix: `docker network rm traefik-public` then `./start.sh` lets Compose recreate it properly.

- Convention: anything that goes in `.env` → Bitwarden. Architecture/docs → Obsidian.
- To recreate a `.pem` from Bitwarden: paste content to file, `chmod 600`.

- Two accounts: `jordimo` (personal), `JordiMartinez-TMA` (corporate).
- SSH keys: `~/.ssh/id_ed25519` (personal), `~/.ssh/github_corporate` (corporate). Named descriptively following existing convention (`do_isidora` pattern).

### Decisions made

- **Repo is public** — contains no secrets (`.env` is gitignored), and making it public avoids needing GitHub auth on the VM. Simplest path.
- **Path-based routing over subdomains** — VM has only an IP, no domain. All projects share the IP and route by path prefix (`/project-name`).
- **No TLS yet** — ports 80/443 pending IT enablement. Traefik config only has a `web` entrypoint on port 80. HTTPS can be added later.
- **Postgres password via `.env` not in repo** — generated with `openssl rand -base64 32`, stored in Bitwarden, manually placed on VM.
- **Corporate GitHub as default on corporate laptop, personal as default on dev laptop** — matches primary use case of each machine, alias for the other.

## Session 2 — Marie deployment on DigitalOcean (isidora)

### What we worked on

After AWS VM issues (needed reboot), pivoted to deploying Marie on the existing DigitalOcean droplet (isidora) to get it accessible tonight. Also added Langfuse to AWS infra, production Dockerfile stages, and `init-project.sh`.

### What got done

**AWS infra updates (before pivot to DO)**

- Added Langfuse to `traefik/docker-compose.yml`. Initially tried path-based routing (`/langfuse`) but Langfuse requires `NEXT_PUBLIC_BASE_PATH` at build time. Switched to direct port 3030, accessible via SSH tunnel. Commits `c98f9eb`, `6d07649`.
- Removed Docker provider from Traefik — was failing with API version mismatch (`client version 1.24 is too old`). All routing uses file provider anyway. Commit `88e9ec8`.

- Created `init-project.sh` — full first-time project setup from corporate laptop: clone, create DB, pause for `.env`, register routing, build, migrate, create admin. Commit `b95dac0`.
- Created comprehensive `README.md` for the infra repo. Commit `a636b7d`.
- Updated `deploy.sh` to use `docker-compose.vm.yml` for projects. Commit `a103e29`.

**Marie production Dockerfiles**

- Added `production` target to both API and web Dockerfiles. Dev stage uses volume mounts for the shared package; production builds `packages/shared` and the app inside the container.
- API production runs `node apps/api/dist/main`, web production uses `serve -s`.
- Fixed missing `tsconfig.base.json` — shared package `tsconfig.json` extends `../../tsconfig.base.json` which wasn't being copied in the Dockerfile. Commit `aafea19`.
- Created `docker-compose.vm.yml` (AWS, path-based) and `docker-compose.do.yml` (DO, host-based with TLS).

**Marie on DigitalOcean (isidora)**

- Isidora (`174.138.33.106`) already runs Traefik v3.6, Postgres 16, Redis 7, plus several other apps (Caitie, newsintel, company-intel, vaultwarden).
- Key differences from AWS: host-based routing with real domains, Docker labels (not file provider), Let's Encrypt TLS, two networks (`caitie-infra_traefik`, `caitie-infra_shared_services`), Postgres user is `caitie_admin_db` (not `postgres`).
- DNS: added `marie.lostriver.llc` A record pointing to `174.138.33.106` in Cloudflare (manually).
- Deploy key: generated `~/.ssh/github_deploy` on isidora, added to `github.com/jordimo/Marie/settings/keys` (read-only). Also had to run `ssh-keyscan github.com >> ~/.ssh/known_hosts` for first connection.
- Created `docker-compose.do.yml` with Traefik Docker labels, Let's Encrypt cert resolver, priority-based routing (API at 200, web at 100).

**Issues encountered and fixed on DO**

- **DB user mismatch**: isidora uses `caitie_admin_db` not `postgres`. Also had to specify `-d caitie_db` since the default DB doesn't match the username.
- **Migrations**: had to run `docker exec -w /app/apps/api marie-api npx drizzle-kit migrate` — `drizzle.config.ts` is in `apps/api/`, not project root.
- **API crash loop**: containers started before migrations ran, causing `42P01` (relation not found) errors. Self-resolved after migrations.
- **Traefik redirect middleware**: `marie-https-redirect@docker` was defined on web container but referenced from API container. Fixed by removing the API redirect router — single redirect on web handles all HTTP→HTTPS.
- **Web container healthcheck**: used `nc -z` but `nc` isn't installed in the `node:22-slim` production image. Healthcheck always failed, causing Traefik to skip routing to the web container entirely. Fixed by using Node's `fetch()` for the healthcheck.
- **Admin user creation**: interactive node scripts over SSH have severe escaping issues. Created `scripts/create-admin.js` that runs inside the API container with interactive prompts. Had to `docker cp` it in since `scripts/` isn't copied in the Dockerfile.

**Tools installed on servers**

- `micro` editor on AWS VM (via `apt`), attempted on DO but `deploy` user has no sudo — used `curl https://getmic.ro | bash` instead.

### Decisions made

- **DO deployment takes priority over AWS** — AWS VM needed reboot, Marie needed to be accessible tonight.
- **Separate compose files per environment** — `docker-compose.yml` (local dev), `docker-compose.vm.yml` (AWS), `docker-compose.do.yml` (DO). Keeps each environment's config independent.
- **Deploy keys over giving VM full GitHub access** — read-only, scoped to one repo. New key per repo, per server.
- **Priority-based routing over negated PathPrefix** — `!PathPrefix(/api)` worked on newsintel but was simpler to reason about with explicit priorities.
- **Healthcheck determines Traefik routing** — if a container's healthcheck fails, Traefik won't route to it. This was the root cause of the web 404.

## Session 3 — Infra unification planning

### What we worked on

After getting Marie live on DO, planned the convergence of all three environments (local, AWS, DO) to unified naming, networking, and project conventions.

### What got done

**Infra unification plan** (`docs/plans/infra-unification.md`)

- Documented current state across all three environments — different networks, container names, routing strategies, compose files, Traefik providers
- Designed target state: one `docker-compose.yml` (dev) + one `docker-compose.prod.yml` (prod) per project, differences live in `.env` only
- Standardized naming: network `infra` everywhere, containers `postgres`/`redis`/`traefik`, projects `{name}-api`/`{name}-web`
- Infra repo rename: `THECOLLECTIVE_AWS_INFRA` → `infra`
- Four-phase migration plan: local (no risk) → AWS (low risk) → DO (careful, backup first) → cleanup
- DO migration detailed with data safety guarantees — pg_dumpall backup, volume preservation, one project at a time, caitie last

**Standard project scripts convention**

- All scripts are `.sh` files run from the host command line
- Container internals (Node, bcrypt, etc.) are implementation details hidden inside `.sh` via `docker exec`
- Every project gets: `scripts/start.sh`, `scripts/stop.sh`, `scripts/create-admin.sh`, `scripts/migrate.sh`, `scripts/setup-env.sh`

**`.env` management design**

- One `.env.example` per project with three sections: environment-specific / secrets / defaults
- `DOMAIN` is the single variable that drives routing, frontend URL, and callback URLs
- Future `scripts/setup-env.sh` to interactively generate `.env` — no more SSH heredoc copy-pasting

### Decisions made

- **Infra repo name:** `jordimo/infra` — server-agnostic
- **AWS stays path-based** for now — no domain, IP-only, VPN access only
- **All DO projects migrate** — marie, caitie, newsintel, company-intel, vaultwarden
- **Scripts are always `.sh`** — user never runs `.js` directly, even if Node runs under the hood
- **DO Postgres user** — keep `caitie_admin_db` for now, migrate to `postgres` later as a separate step after everything else is stable
