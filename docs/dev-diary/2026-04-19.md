## Session 14 — Langfuse v2 → v3 migration (local + aws01)

### What we worked on

Started as a cost-tracking gap in Betty's Langfuse dashboard — `totalCost: 0` on `gpt-5.4*` generations because the seeded model catalog in `langfuse/langfuse:2` pre-dates March 2026. Pulled on the thread, evaluated whether to add custom model entries vs. upgrade to v3, and decided that "beginning of both projects" is the cheapest time to pay the v3 migration cost. Migrated local and aws01 end-to-end; left Isidora for a follow-up session on the machine with the DO SSH key.

### Why upgrade, not add custom models

Adding a custom model in the v2 UI would have solved the immediate problem in 5 minutes. Trade-off:

- Langfuse v2 shipped its last patch in Nov 2025 — one release in five months vs. ~100 v3 releases in the same window. v2 is EOL-track even though no formal sunset date is published.
- v2 → v3 is only a supported upgrade between **v2.92.x ↔ v3.29.x**. Newer v3 releases drop the shim code. Skipping the bridge becomes an export/reimport later.
- Migration difficulty scales with trace volume, not project age. Local had 22 observations; aws01 had 4. Minutes of backfill at this scale, potentially hours later.
- v3-only features (LLM-as-judge, prompt experiments, seeded prices for every new OpenAI release) will matter more as Betty + Marie grow.

Decided the migration front-loads cost that gets strictly more expensive with time.

### What got done

**Compose refactor (`214f835`, `9d7ecd2`):**
- Replaced the single `langfuse:2` container with four services: `langfuse-web` + `langfuse-worker` (both pinned via `LANGFUSE_IMAGE_TAG`), `clickhouse` (trace storage), `minio` (event + media blobs). All on the shared `infra` network.
- Reused existing shared `postgres` (transactional data: users, orgs, projects, prompts, API keys — stays in pgvector) and shared `redis` (queue + cache). No duplication.
- Added a network alias so `langfuse-web` also resolves as `langfuse`. Existing projects pointing at `http://langfuse:3000` keep working with zero env-file edits. Drop the alias once all consumers explicitly migrate to `langfuse-web`.
- Kept Traefik labels on `langfuse-web` so `langfuse.local` routing on local (and future DNS-backed routing) works unchanged.
- Added `REDIS_AUTH` passthrough after aws01 (which runs Redis with `--requirepass`) needed it. Empty default keeps local (no auth) working.

**Secrets added to each `.env`:**

- `CLICKHOUSE_USER`/`CLICKHOUSE_PASSWORD`, `MINIO_ROOT_USER`/`MINIO_ROOT_PASSWORD` — fresh per environment (local + aws01 got their own values).
- `LANGFUSE_IMAGE_TAG` — pinned to `3.29.0` for the initial migration boot, then flipped to `3` once the worker log reports "No background migrations to run".

### Decisions

- **Bridge through v3.29.0 before bumping to `:3`.** The docs are explicit about the narrow compatibility window. Splitting the upgrade into two boots (v3.29.0 for backfill → `:3` for current) costs one extra `docker compose up -d` and buys a safe rollback point.
- **Pin `LANGFUSE_IMAGE_TAG` via env, not docker-compose.yml.** Same compose across environments; the tag differs per-server during rollout. Lets us roll forward on local while leaving Isidora on the bridge version if needed.
- **Network alias for `langfuse` rather than editing every project's `.env`.** Zero-change consumer migration. Will be removed when Betty + Marie explicitly cut over to `langfuse-web` in their own configs.
- **No formal maintenance window for aws01.** Marie was idle and the full cutover (stop v2 → start clickhouse/minio → start v3 → backfill → bump) was under 5 minutes. Visible outage was a handful of seconds where `http://langfuse:3000` resolved to nothing.

### Issues encountered

- **Invented a fake ClickHouse digest.** Pasted `sha256:c22c6e4e…` into the compose that doesn't exist in the registry — `docker compose pull` returned `manifest unknown`. Fixed by dropping the digest and pinning to tag `25.3`. Worth pinning a real digest in a follow-up once the rollout stabilizes.
- **First local smoke test looked like it failed.** Sent a test trace via a one-shot `docker exec … node -e …` script that exited before the Langfuse SDK flushed its internal buffer. No v3-smoke-test trace appeared for minutes. Fixed by adding `await lf.shutdownAsync()` at the end of the script. The live Betty API doesn't have this problem — it stays running — so the finding was specific to ad-hoc smoke scripts.
- **aws01 Redis needs `--requirepass`.** Local Redis doesn't. First aws01 boot of `langfuse-web` on the exact compose used locally would have failed to authenticate. Caught during reconnaissance (before touching anything on aws01); patched the compose with a `REDIS_AUTH` passthrough and pushed that fix before executing on aws01.
- **Post-migration, old traces still show `totalCost: 0`.** Cost is computed at generation time against the seeded catalog. Generations written under v2 (when the catalog didn't know about `gpt-5.4*`) stay at 0 forever; only new generations get priced. Expected behavior; noted because it can confuse someone expecting retroactive pricing.
